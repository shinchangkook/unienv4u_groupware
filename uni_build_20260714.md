# unienv4u 그룹웨어 실제 구축 가이드

- **작성일**: 2026-07-14
- **대상**: `index.html` (v0.9.33, 단일 파일 SPA)
- **목적**: 데모(localStorage) 수준의 프로토타입을 **실제 다중 사용자 운영 그룹웨어**로 전환하기 위한 구축 방안 정리
- **회사**: 유앤아이환경기술(주)

---

## 0. 요약 (TL;DR)

현재 `index.html`은 **프론트엔드 완성도는 높지만 서버·DB·인증이 전혀 없는** 브라우저 단독 데모입니다.
실제 그룹웨어가 되려면 아래 3가지가 반드시 추가되어야 합니다.

1. **서버 + 데이터베이스** — 여러 사람이 같은 데이터를 공유 (현재는 각자 브라우저에만 저장됨)
2. **제대로 된 인증/인가** — 서버 세션·토큰, 비밀번호 해시 (현재는 평문 저장)
3. **배포 인프라** — 도메인, HTTPS, 백업 (현재는 파일 열기)

**권장 경로**: 프론트엔드(HTML/CSS)는 최대한 재활용하고, 백엔드로 **Supabase(PostgreSQL + Auth + Storage)** 를 붙이는 방식이 가장 빠르고 비용 효율적입니다. 규모가 커지거나 사내 서버 정책이 있으면 **Spring Boot / Node + MySQL** 전통 스택으로 갑니다.

전환 규모는 **개발자 1명 기준 약 2~4개월**(MVP), 전체 기능 이관까지 **4~6개월**로 추정합니다.

### 1.3 "실제 그룹웨어"로서 결정적 결함
프로토타입이라 당연하지만, 운영에 앞서 반드시 알아야 할 한계:

1. **데이터가 공유되지 않음** — A의 브라우저와 B의 브라우저 데이터가 완전히 별개. 전자결재·게시판·계약 등 협업 기능이 근본적으로 성립 불가.
2. **데이터 영속성 없음** — 브라우저 캐시 삭제, 기기 변경, 시크릿 모드 시 전부 소실. 백업 불가.
3. **심각한 보안 문제**
   - 비밀번호가 `uni_members`에 **평문**으로 저장 (`m.pw===pw` 대조).
   - 주민번호를 `btoa()`로 "간단 암호화" — 이는 **암호화가 아니라 인코딩**이며 누구나 즉시 복호화 가능. 개인정보보호법 위반 소지.
   - 인증·권한 체크가 전부 클라이언트에서 이뤄져 개발자도구로 우회 가능.
4. **동시성/무결성 없음** — 트랜잭션, 유효성 검증, 중복 방지 로직이 서버에 없음.
5. **감사 추적 없음** — 누가 언제 무엇을 승인/수정했는지 서버 기록이 없음(전자결재·인사에 필수).
6. **용량 한계** — localStorage는 약 5~10MB. 성적서·첨부파일이 쌓이면 한계 도달.

> 결론: 현재 파일은 **훌륭한 화면 설계도(=프론트 명세)** 로 활용하고, 그 위에 서버·DB를 새로 구축하는 것이 목표입니다.

## 2. 목표 아키텍처 (To-Be)

```
[브라우저 - SPA]
  index.html 기반 UI (기존 화면 재활용)
        │  HTTPS / REST(JSON)
        ▼
[백엔드 API 서버]
  인증(JWT/세션) · 인가(RBAC) · 비즈니스 로직 · 유효성 검증
        │
        ▼
[데이터베이스 - PostgreSQL/MySQL]     [파일 스토리지]      [메일/알림]
  회원·계약·성적서·휴가 등              성적서 PDF·첨부      결재 알림 메일
```

핵심 원칙:
- **모든 권한·유효성 검증은 서버에서** (클라이언트 검증은 UX 보조일 뿐).
- **localStorage는 캐시/임시 용도로만**, 원본(source of truth)은 항상 서버 DB.
- 기존 화면/CSS는 그대로, `localStorage.getItem/setItem` 호출부만 **API 호출(fetch)** 로 치환.

## 3. 기술 스택 선택

### 방안 A (권장) — Supabase 백엔드 + 기존 프론트 점진 이관
소규모 팀·빠른 구축에 최적. 서버 코드를 거의 안 짜도 됨.

| 계층 | 기술 | 역할 |
|------|------|------|
| 프론트 | 기존 HTML/CSS/JS 유지 → 이후 필요 시 Vite/React 이관 | 화면 |
| 인증 | **Supabase Auth** | 이메일/비번, 세션, 승인 플로우 |
| DB | **Supabase PostgreSQL** | 11개 도메인 → 테이블 |
| 권한 | **Row Level Security(RLS)** | 부서·마스터 권한을 DB 레벨에서 강제 |
| 파일 | **Supabase Storage** | 성적서/첨부 |
| 실시간 | Supabase Realtime | 결재·게시판 실시간 반영(선택) |
| 배포 | Vercel / Netlify / Cloudflare Pages | 정적 프론트 호스팅 |

**장점**: 백엔드 개발 최소화, 인증·파일·권한이 기본 제공, 무료 티어로 시작 가능.
**단점**: 사내 온프레미스 정책이 있으면 부적합, 벤더 종속.

### 방안 B — 전통 3-Tier (한국 SI 표준, 온프레미스 가능)
사내 서버 운영·기존 인프라 활용·규모 확장에 유리.

| 계층 | 기술 (택1) |
|------|-----------|
| 프론트 | 기존 HTML → 또는 React/Vue SPA 재작성 |
| 백엔드 | **Spring Boot(Java)** 또는 **NestJS/Express(Node)** |
| DB | **MySQL 8** 또는 **PostgreSQL** |
| 인증 | Spring Security + JWT / Passport |
| 파일 | 서버 디스크 또는 S3/MinIO |
| 배포 | Docker + Nginx (사내 서버 / 클라우드 VM) |

**장점**: 완전한 통제, 온프레미스 가능, 확장·커스터마이징 자유.
**단점**: 개발·운영 공수 큼(백엔드 전면 개발 필요).

### 권장 결론
- **1인/소수 개발, 빠른 오픈 우선** → **방안 A (Supabase)**
- **사내 IT 정책상 자체 서버 필수, 장기 확장** → **방안 B (Spring Boot + MySQL)**

이 문서의 이후 예시는 **방안 A**를 기준으로 하되, 개념은 방안 B에도 동일 적용됩니다.

## 4. 데이터베이스 설계 (localStorage → 테이블)

localStorage의 11개 JSON 도메인을 관계형 테이블로 정규화합니다. 아래는 핵심 테이블 스케치(PostgreSQL 기준).

```sql
-- 회원 (uni_members)  ※ 비밀번호는 저장하지 않음 → Auth가 관리
create table members (
  id            uuid primary key default gen_random_uuid(),
  email         text unique not null,
  name          text not null,
  ename         text,
  dept          text,               -- 부서 (normalizeDept 규칙 적용)
  rank          text,               -- 직급
  tel           text,
  jumin_enc     text,               -- 주민번호: 서버측 강암호화(AES) 또는 저장 지양
  hire_date     date,
  leave_date    date,               -- 퇴사일
  status        text default 'pending',   -- pending/active/rejected
  is_admin      boolean default false,    -- 마스터
  created_at    timestamptz default now()
);

-- 메뉴 권한 (uni_menu_perms)
create table menu_perms (
  menu_key    text primary key,     -- emp, env, contract ...
  dept_allow  text[],               -- 접근 허용 부서
  submasters  uuid[]                -- 부마스터 지정
);

-- 거래처 (uni_clients)
create table clients (
  id          uuid primary key default gen_random_uuid(),
  name        text not null,
  biz_no      text,                 -- 사업자번호
  ceo         text, addr text, tel text, email text,
  memo        text,
  created_at  timestamptz default now()
);

-- 통합 계약 (uni_ctu)
create table contracts (
  id          uuid primary key default gen_random_uuid(),
  client_id   uuid references clients(id),
  title       text,
  is_joint    boolean default false,   -- 공동계약
  amount      numeric,
  start_date  date, end_date date,
  status      text,
  created_at  timestamptz default now()
);

-- 청구/수금 (uni_billing)
create table billings (
  id          uuid primary key default gen_random_uuid(),
  contract_id uuid references contracts(id),
  client_id   uuid references clients(id),
  bill_date   date, amount numeric,
  paid_amount numeric default 0,
  kind        text,                    -- 청구/수금
  is_bad_debt boolean default false,   -- 대손
  created_at  timestamptz default now()
);

-- 환경질 성적서 (uni_env) + 관리번호 (uni_det_mno)
create table env_reports (
  id          uuid primary key default gen_random_uuid(),
  mno         text unique,             -- 관리번호
  client_id   uuid references clients(id),
  contract_id uuid references contracts(id),
  category    text,                    -- 수질/대기/토양/소음
  payload     jsonb,                   -- 폼 상세(과도 정규화 회피)
  locked      boolean default false,
  approval_status text,
  created_by  uuid references members(id),
  created_at  timestamptz default now()
);

-- 휴가 (uni_leaves) / 연차 (uni_annual) / 설정 (uni_lv_settings)
create table leaves (
  id          uuid primary key default gen_random_uuid(),
  member_id   uuid references members(id),
  type        text,                    -- 연차/반차/병가...
  start_date  date, end_date date, days numeric,
  status      text default 'pending',  -- 결재 상태
  approver_id uuid references members(id),
  reason      text,
  created_at  timestamptz default now()
);
create table annual_leaves (
  member_id   uuid references members(id),
  year        int,
  granted     numeric, used numeric, remaining numeric,
  primary key (member_id, year)
);
create table leave_settings ( key text primary key, value jsonb );

-- 메모 로그 (uni_memos)  ※ 여러 도메인에서 공용
create table memos (
  id          uuid primary key default gen_random_uuid(),
  target_type text,                    -- env/contract/client...
  target_id   text,                    -- memoKey 대응
  author_id   uuid references members(id),
  body        text,
  created_at  timestamptz default now()
);

-- (신규 권장) 전자결재 문서/라인, 감사 로그
create table approvals ( ... );        -- 기안→팀장→대표 라인
create table audit_logs (
  id bigserial primary key,
  actor_id uuid, action text, target text,
  detail jsonb, created_at timestamptz default now()
);
```

설계 메모:
- **성적서(`env_reports.payload jsonb`)** 처럼 필드가 많고 가변적인 폼은 JSONB로 두면 초기 이관이 빠릅니다. 조회 조건이 되는 핵심 필드(관리번호·거래처·카테고리)만 컬럼으로 승격.
- **주민번호**: 원칙적으로 저장하지 않는 것을 권장. 꼭 필요하면 서버 전용 키의 **AES 강암호화 + 접근 로그**. `btoa`는 즉시 폐기.
- 모든 테이블에 `created_at`/`updated_at`, 변경 이력은 `audit_logs`로.

## 5. 인증 / 인가 재설계

현재(`doLogin`)의 문제 → 서버 기반으로 전환:

| 항목 | 현재 (index.html) | 목표 |
|------|-------------------|------|
| 비밀번호 | 평문 저장·비교 | 해시(bcrypt/argon2) 또는 Auth 서비스 위임 |
| 로그인 | 클라이언트에서 대조 | 서버 검증 → 세션/JWT 발급 |
| 승인 대기(`pending`) | 클라 플래그 | DB status + 관리자 승인 API |
| 마스터/부마스터 | 클라 플래그·`canAccess` | 서버 미들웨어 + DB RLS |
| 부서별 메뉴 권한 | `MENU_PERMS` 클라 | 서버 권한 테이블 + 매 요청 검증 |

핵심: **`canAccess`/`deptCanAccess` 같은 클라 권한 함수는 UI 노출 제어용으로만 남기고, 실제 데이터 접근은 서버가 재검증**해야 합니다(클라 검증은 우회 가능).

Supabase 사용 시 예:
```js
// 회원가입 (승인 대기 상태로)
const { data, error } = await supabase.auth.signUp({ email, password });
// members 테이블에 status='pending' 프로필 생성 → 마스터가 승인
```
RLS 정책 예:
```sql
-- 본인 또는 관리자만 회원 조회
create policy "member read" on members for select
  using ( auth.uid() = id or exists(
    select 1 from members me where me.id = auth.uid() and me.is_admin));
```

## 6. 프론트엔드 이관 전략

**기존 화면을 버리지 않고 재활용**하는 것이 이 프로젝트의 최대 장점입니다.

### 6.1 데이터 접근 계층 추상화 (1단계, 가장 중요)
현재 코드 곳곳에 흩어진 `localStorage.getItem('uni_xxx')` / `setItem`을 **단일 저장소 모듈**로 감쌉니다.

```js
// 현재 (직접 접근, 110곳)
memberDB = JSON.parse(localStorage.getItem('uni_members')||'[]');

// 1단계: 어댑터로 감싸기 (내부는 아직 localStorage)
const store = {
  async list(domain){ return JSON.parse(localStorage.getItem('uni_'+domain)||'[]'); },
  async save(domain, rows){ localStorage.setItem('uni_'+domain, JSON.stringify(rows)); },
};

// 2단계: 어댑터 내부만 API로 교체 (화면 코드는 그대로)
const store = {
  async list(domain){ const r = await fetch(`/api/${domain}`); return r.json(); },
  async save(domain, row){ return fetch(`/api/${domain}`, {method:'POST', body:JSON.stringify(row)}); },
};
```
이렇게 하면 183개 함수의 화면 로직을 거의 손대지 않고, **어댑터 한 곳만 교체**하여 서버 연동이 됩니다. (동기→비동기 전환에 따라 `await` 추가 작업은 필요)

### 6.2 이후(선택) 프레임워크 이관
운영이 안정되면 유지보수를 위해 **Vite + React/Vue**로 화면을 컴포넌트화하는 것을 권장하나, 필수는 아닙니다. 단일 파일이 5,350줄을 넘어 유지보수가 어려워질 때 착수.

## 7. 단계별 로드맵 (권장 실행 순서)

### Phase 1 — 백엔드 기반 (2~3주)
- [ ] DB 스키마 생성(4장), 마이그레이션 스크립트

## 8. 배포 / 운영 인프라

| 항목 | 방안 A (Supabase) | 방안 B (자체 서버) |

**연동 방식(전환 리스크 최소화)**: 5,000줄 화면 코드를 거의 손대지 않기 위해
① **로그인 시 하이드레이트**(서버 우선, 서버가 비면 로컬 시드를 최초 이관) +
② **`localStorage.setItem` 후크로 쓰기 미러링**(대상 키 저장 시 서버로 디바운스 반영)을 사용.
각 화면의 렌더 함수는 기존처럼 localStorage 를 읽으므로 수정 불필요. (향후 화면별로 명시적
`UniStore` 호출로 순차 치환 가능.)

**검증 완료(브라우저 실측)**:
- 인증: 로그인(정상/오류/승인대기), 회원가입→관리자 승인(사번 자동 부여)→로그인, 무토큰 API 401.
- Phase 2: 빈 서버에 최초 로그인 시 시드 **자동 이관(거래처 19 · 환경 5)**, 거래처 추가 시 서버 반영,
  중첩 필드(pays/trips) 무손실 왕복, 필수값 누락 시 400 검증, det_mno KV 반영,
  로컬 삭제 후 하이드레이트로 서버에서 복원(크로스 세션 공유) 확인. 콘솔 에러 없음.
